# SKILL: QA、平衡与测试

## 使用场景

当任务涉及功能验收、测试用例、平衡参数、性能检查、微信真机测试或发布准备时，阅读本文件。

---

## 1. 测试目标

这款游戏最重要的测试不是“能不能赢”，而是：

1. 找人者能否通过记忆和观察做判断；
2. 躲藏者能否通过合理伪装获得成就感；
3. 有限拍击次数是否制造压力；
4. 手机操作是否足够顺手；
5. 联机同步是否可信。

---

## 2. 基础功能测试

每个阶段都应至少测试：

- 阶段能正确进入和退出；
- 倒计时正确；
- 输入在合法阶段生效；
- 输入在非法阶段被忽略；
- UI 与实际状态一致；
- 重开一局不会保留错误状态。

---

## 3. 核心玩法测试用例

### Preview

- [ ] 玩家角色不可见。
- [ ] 双方不能移动。
- [ ] 地图道具全部可见。

### Hide

- [ ] 找人者黑屏或看到静态屏保。
- [ ] 找人者不能操作。
- [ ] 躲藏者能正常速度移动。
- [ ] 躲藏者停下后自动变成道具。
- [ ] 躲藏者能无冷却切换道具。

### Seek

- [ ] 找人者能移动。
- [ ] 找人者能朝最后移动方向拍击。
- [ ] 扇形范围内多个普通道具会被打碎。
- [ ] 扇形范围内躲藏者会被抓。
- [ ] 躲藏者保持道具形态慢速移动。
- [ ] 躲藏者能无冷却切换道具。
- [ ] 拍击次数用完后本轮结束。

### Scoring

- [ ] 抓 1 人 +1。
- [ ] 全抓额外 +1。
- [ ] 存活躲藏者 +1。
- [ ] 被抓躲藏者 +0。
- [ ] 每人轮流当找人者。

---

## 4. 平衡观察指标

每次 playtest 记录：

```text
玩家数
地图
平均每轮搜索时长
找人者平均抓获人数
找人者拍击用完比例
躲藏者移动次数
躲藏者切换道具次数
找人者误拍次数
玩家主观评价
```

---

## 5. 推荐平衡目标

4 人局，3 名躲藏者：

- 找人者平均抓到 1~2 人比较健康。
- 全抓应该发生，但不能太频繁。
- 拍击次数用完导致结束应经常出现。
- 躲藏者完全不动也可能赢，但不是唯一策略。
- 当着找人者面切换道具应该高风险。

---

## 6. 真机与网络测试

至少测试：

- 两台手机同 Wi-Fi；
- 两台手机不同网络；
- 弱网或高延迟；
- 玩家中途断线；
- 找人者断线；
- 分享进房；
- 一局结束后再开一局。

---

## 7. 发布前检查

- [ ] 无明显崩溃。
- [ ] 首包体积记录。
- [ ] WebSocket 地址配置正确。
- [ ] 隐私与用户信息处理无明显问题。
- [ ] 资源授权记录完整。
- [ ] 没有调试作弊 UI 暴露给普通玩家。
- [ ] 至少完成 10 场 4 人测试。

---

## 8. 禁止事项

- 不要只在编辑器里测试后就认为完成。
- 不要忽略找人者和躲藏者双方体验差异。
- 不要为了让某一方赢而加入复杂补偿技能。
- 不要在没有数据的情况下大幅改核心参数。
