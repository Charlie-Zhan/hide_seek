# SKILL: 微信小游戏适配

## 使用场景

当任务涉及微信小游戏构建、登录、分享、房间码、WebSocket、包体、适配、性能或发布检查时，阅读本文件。

---

## 1. 微信小游戏产品原则

- 第一版优先做好友房，不做随机匹配。
- 使用微信分享进入房间。
- 操作必须适合触屏：左轮盘 + 右侧单按钮。
- 单局时长要短，推荐 4 人整场控制在 4~6 分钟。
- UI 文案要轻量，避免复杂教程。

---

## 2. 构建目标

阶段目标：

1. Cocos 项目能构建到微信小游戏平台。
2. 微信开发者工具可打开并运行。
3. 能通过 WebSocket 连接测试服务器。
4. 能创建房间、分享房间码、好友进入房间。
5. 首包资源可控，非必要资源延迟加载。

---

## 3. 登录与身份

MVP 可先使用临时身份：

```text
playerId = openid 或开发环境临时 uuid
nickname = 微信昵称或默认名
avatar = 微信头像或默认头像
```

注意：

- 不要把用户敏感信息写入日志。
- 服务端以 `playerId` 识别玩家。
- 房间内显示昵称即可，不依赖复杂账号系统。

---

## 4. 分享进房

推荐分享参数：

```text
roomId=ABC123
```

流程：

1. 房主创建房间。
2. 客户端调用分享。
3. 好友点击分享卡片。
4. 客户端读取 query 中的 roomId。
5. 自动进入对应房间。

---

## 5. WebSocket 注意事项

- 使用 `wss`。
- 服务端需要心跳。
- 客户端断线后尝试重连。
- 重连成功后发送 `resume_room` 或 `join_room`。
- 所有关键状态以服务端为准。

---

## 6. 包体与资源

- 首包只放：启动、登录、房间、第一张地图、基础 UI、核心角色与道具。
- 后续地图、皮肤、音效可使用分包或远程资源。
- Kenney 原始大图不要全部塞进首包。
- 使用图集和压缩纹理。
- 音频文件尽量短。

---

## 7. 屏幕适配

必须验证：

- 竖屏还是横屏，MVP 推荐横屏。
- 刘海屏安全区域。
- 左轮盘不被系统手势区影响。
- 右侧按钮足够大。
- 小屏设备上 HUD 不遮挡核心地图。

---

## 8. 微信阶段验收清单

- [ ] 微信开发者工具可运行。
- [ ] 创建房间成功。
- [ ] 分享卡片携带房间参数。
- [ ] 另一个客户端可进入房间。
- [ ] WebSocket 可连接并断线重连。
- [ ] 搜索阶段触摸输入稳定。
- [ ] 首包体积有记录。
- [ ] 基础错误提示可见。

---

## 9. 禁止事项

- 不要依赖浏览器 DOM API。
- 不要在微信环境中使用不可用的 Node API。
- 不要把服务器地址硬编码在多个脚本里。
- 不要为了上线前演示引入复杂登录系统。
